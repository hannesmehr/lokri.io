# i18n Error-Konventionen

Developer-Konventionen für die laufende i18n-Innenschicht-Migration.

## Grundsatz

- Backend wirft **Error-Codes**, nicht freie user-facing Texte.
- API-Responses serialisieren strukturierte Fehler über `codedApiError()`.
- Textliche Fallback-Messages werden nicht mehr mitserialisiert. Die UI mappt `details.code` auf lokalisierte Texte und fällt bei unbekannten Codes auf generische Fehlermeldungen zurück.

## Response-Format

Für migrierte Fehler gilt:

```json
{
  "details": {
    "code": "session.accountDisabled",
    "status": 403
  }
}
```

- `details.code` ist der primäre, stabile Schlüssel.
- `details.status` bleibt für UI-Branches und Debugging erhalten.

## Backend-Regel

- Neue non-admin Fehler immer strukturiert serialisieren, bevorzugt via `codedApiError()`.
- Wenn ein Fehler über `ApiAuthError` läuft, gilt dieselbe Regel: `code` ist primär.
- Freie, direkt sichtbare Backend-Texte in non-admin Pfaden vermeiden.

## Frontend-Regel

- Frontend mappt `details.code` auf `t('errors.api.{area}.{key}')`.
- Wenn `details.code` unbekannt ist, fällt die UI auf einen generischen Fehlertext zurück.
- UI-Migrationen passieren bereichsweise in Phase 2; Backend und UI werden bewusst entkoppelt.
- Nachträgliche Retrofits wie `embeddingKey` folgen derselben Regel und nutzen dieselben Error-/Toast-Patterns wie Storage-Settings und Billing.

## Admin-Scope

- Admin-Scope bleibt deutsch.
- Gilt für `app/(admin)/*`, `app/api/admin/*`, `lib/admin/*` und bewusst admin-spezifische Guard-Messages.
- Hintergrund und Ausnahmen sind in [docs/I18N_AUDIT.md](/Users/hannes/Development/private/lokri.io/docs/I18N_AUDIT.md:1) dokumentiert.

## Neue Errors hinzufügen

1. Backend-Code definieren oder bestehenden stabilen Code verwenden.
2. Key in `messages/de.json` und `messages/en.json` anlegen.
3. Response strukturiert serialisieren.
4. Falls die UI den Code schon kennt: auf `details.code` mappen. Falls nicht: generischen Fehlertext beibehalten.
